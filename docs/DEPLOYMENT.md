# Katiba OS deployment runbook

This runbook deploys the Node API to Render and the React PWA to Cloudflare Pages. Flutter clients use the same Render API.

## Recommended presentation architecture

```text
Cloudflare Pages (React PWA) ─┐
Flutter apps ─────────────────┼─ HTTPS ─ Render Free Node API ─ temporary SQLite
                             └───────── OpenAI API (server-side only)
```

The checked-in Blueprint uses Render's Free web-service plan for a zero-cost hackathon demonstration. SQLite runs at `/tmp/katiba-os.db`, so newly created claims and audit events are temporary and disappear whenever the service sleeps, restarts, or redeploys. The seeded demonstration case is recreated automatically at startup.

Before accepting real or persistent pilot data, upgrade the service to a paid instance, attach a disk at `/var/data`, and change `DATABASE_PATH` to `/var/data/katiba-os.db`. For multi-instance or production scale, migrate the same normalized schema to managed PostgreSQL.

PostgreSQL is the preferred production database because it supports connection pooling, replicas, migrations, backups, and horizontal API scaling. Cloudflare D1 is not the preferred database for this architecture: the authoritative API is already on Render, so using D1 would add a Worker/database boundary to every legal-data request.

## 1. Render API

The repository includes `render.yaml`.

1. Push the repository to GitHub or GitLab.
2. In Render, create a Blueprint and select this repository.
3. Keep the service name `katiba-os-api` or update the public URLs in the Cloudflare and Flutter configuration.
4. Set the two secret/place-specific values Render marks as unsynchronized:
   - `OPENAI_API_KEY`: project-scoped OpenAI API key.
   - `ALLOWED_ORIGINS`: comma-separated exact frontend origins, for example `https://katiba-os.pages.dev,https://app.katibaos.example`.
5. Deploy and verify `https://<service>.onrender.com/api/health`.

Render environment variables:

| Variable | Required | Purpose |
|---|---:|---|
| `SESSION_SECRET` | Yes | HMAC signing secret; generated by the Blueprint |
| `DATABASE_PATH` | Yes for SQLite | `/tmp/katiba-os.db` in the Free demo; data is ephemeral |
| `ALLOWED_ORIGINS` | Yes | Exact Cloudflare/custom origins; no wildcard |
| `OPENAI_API_KEY` | Yes for live AI/voice | Kept only on Render |
| `OPENAI_MODEL` | Yes | Structured case analysis; default `gpt-5.4-mini` |
| `OPENAI_TRANSCRIBE_MODEL` | Yes | Voice-to-text; default `gpt-4o-transcribe` |
| `OPENAI_TTS_MODEL` | Yes | Text-to-speech; default `tts-1` |
| `OPENAI_TTS_VOICE` | Yes | Built-in synthetic voice; default `alloy` |
| `NODE_VERSION` | Yes | `24.14.0` for the tested Node SQLite runtime |
| `PORT` | No | Render supplies this automatically |

The API binds to `0.0.0.0` and reads Render's `PORT` automatically.

Free services sleep after inactivity and may take about a minute to wake. Open `/api/health` shortly before presenting and keep the demo focused on the seeded case.

## 2. Cloudflare Pages PWA

Cloudflare Pages settings:

| Setting | Value |
|---|---|
| Framework preset | React (Vite) |
| Build command | `npm ci && npm run build` |
| Build output directory | `dist` |
| Root directory | repository root |
| Node version | `24.14.0` |

Add one public build variable:

```text
VITE_API_URL=https://<your-render-service>.onrender.com
```

Do not add `OPENAI_API_KEY` to Cloudflare Pages. Vite variables are compiled into browser JavaScript and are public.

The included `public/_redirects` provides SPA routing. `public/_headers` adds baseline browser security headers. If the Render hostname changes, update both `VITE_API_URL` and the hostname in `public/_headers`.

## 3. Flutter

Local Android emulator:

```powershell
flutter run --dart-define=KATIBA_API_URL=http://10.0.2.2:8787
```

Production build:

```powershell
flutter build apk --release --dart-define=KATIBA_API_URL=https://<your-render-service>.onrender.com
flutter build web --release --dart-define=KATIBA_API_URL=https://<your-render-service>.onrender.com
```

Microphone permissions are declared for Android, iOS, and macOS. Browser voice capture requires HTTPS, which Cloudflare Pages provides. The UI labels playback as an AI voice; OpenAI-generated audio must not be presented as a human recording.

## 4. Deployment smoke test

Run these checks after every deployment:

1. `/api/health` returns `status: ok` and `mode: live-ai`.
2. Sign in as claimant, paralegal, and lawyer.
3. Create a consented claim and verify it works during the active presentation session.
4. Analyze the claim and confirm `aiMode: openai`.
5. Record a short English and Kiswahili intake statement and verify transcription.
6. Play the AI-generated case summary.
7. Generate a PDF pack and confirm the audit event appears.
8. Confirm a claimant cannot approve a case and a lawyer can.
9. Confirm an unlisted browser origin is rejected by CORS.

## Production migration gate

Before real client data, complete all of the following: PostgreSQL migration, object storage for evidence bytes, encryption/key management, malware scanning, backups/restore test, rate limits, real identity/OTP, DPIA, retention policy, incident response, legal-source editorial workflow, and a supervised pilot with Kenyan counsel.
